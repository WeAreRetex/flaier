# CLI Commands

All commands below are shown from the repo root via `pnpm` scripts.

## scaffold

Generate a starter flow spec from a template.

```bash
pnpm agents:scaffold -- --title "Checkout Flow" --template branching --out ./apps/viewer/flow-specs/checkout.flow.json
```

### Options

| Flag                   | Description                              | Default                         |
| ---------------------- | ---------------------------------------- | ------------------------------- |
| `--title <text>`       | Flow title (required)                    | —                               |
| `--description <text>` | Flow subtitle                            | —                               |
| `--template <name>`    | `linear`, `branching`, or `architecture` | `linear`                        |
| `--language <name>`    | CodeNode syntax language                 | `typescript`                    |
| `--out <path>`         | Output file path                         | `./flow-specs/<slug>.flow.json` |
| `--force`              | Overwrite existing file                  | —                               |

### Examples

```bash
# Linear flow with a custom output path
pnpm agents:scaffold -- --title "User Registration" --out ./apps/viewer/flow-specs/user-reg.flow.json

# Branching flow for a validation step
pnpm agents:scaffold -- --title "Webhook Dispatch" --template branching --language typescript

# Architecture diagram for the Slidev example app
pnpm agents:scaffold -- --title "Payment System" --template architecture --out ./apps/slides/flow-specs/payment-arch.flow.json
```

## validate

Validate a spec or manifest against the schema and flaier readiness rules.

```bash
pnpm agents:validate -- ./apps/viewer/flow-specs/manifest.json
```

Validation checks:

- JSON schema compliance (`@json-render/core` schema)
- All `children` IDs reference existing elements
- All `transitions[].to` IDs reference existing elements
- `FlowTimeline` is the root element
- Required props are present on each node type

When a manifest contains remote `http(s)` sources, validation skips fetching those remote specs and validates the local ones it can read.

### Options

| Flag     | Description |
| -------- | ----------- |
| `--help` | Show usage  |

## manifest

Scan a directory for `*.flow.json` files and write a manifest.

```bash
pnpm agents:manifest -- --dir ./apps/viewer/flow-specs --out ./apps/viewer/flow-specs/manifest.json
```

### Options

| Flag                | Description                                          | Default                 |
| ------------------- | ---------------------------------------------------- | ----------------------- |
| `--dir <path>`      | Directory to scan                                    | `flow-specs`            |
| `--out <path>`      | Output manifest path                                 | `<dir>/manifest.json`   |
| `--default <id>`    | Preferred `defaultFlowId`                            | First discovered flow   |
| `--skip-validate`   | Skip schema and readiness checks before writing      | `false`                 |

The generated manifest derives each flow's `id` from the filename (without `.flow.json`) and `title` from the spec's `FlowTimeline.props.title`. Existing manifest entries are preserved; new files are appended.
