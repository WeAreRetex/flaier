# Multi-Flow Manifest

A manifest is a JSON index of multiple flow specs. When `<FlowNarrator>` loads a manifest, it shows a dropdown so users can switch between flows.

## Manifest format

```json
{
  "version": 1,
  "defaultFlowId": "auth-login",
  "flows": [
    {
      "id": "auth-login",
      "title": "Auth Login",
      "description": "Request to session issuance",
      "src": "./auth-login.flow.json",
      "tags": ["auth", "api"],
      "entrypoints": ["apps/web/server/api/login.post.ts"]
    },
    {
      "id": "billing-retry",
      "title": "Billing Retry",
      "description": "Failed charge recovery loop",
      "src": "./billing-retry.flow.json",
      "tags": ["billing", "async"]
    }
  ]
}
```

| Field           | Type                         | Description                           |
| --------------- | ---------------------------- | ------------------------------------- |
| `version`       | `number`                     | Manifest schema version. Use `1`.     |
| `defaultFlowId` | `string`                     | ID of the flow shown on initial load. |
| `flows`         | `FlowNarratorManifestFlow[]` | List of flow entries.                 |

### Flow entry fields

| Field         | Type                         | Description                                      |
| ------------- | ---------------------------- | ------------------------------------------------ |
| `id`          | `string`                     | Required. Unique identifier.                     |
| `title`       | `string`                     | Display name in the flow picker dropdown.        |
| `description` | `string`                     | Short description shown in the picker.           |
| `src`         | `string \| FlowNarratorSpec` | Path to the spec file, or an inline spec object. |
| `tags`        | `string[]`                   | Metadata tags for filtering.                     |
| `entrypoints` | `string[]`                   | Source files this flow originates from.          |

## Recommended on-disk layout

```
flow-specs/
  manifest.json
  auth-login.flow.json
  billing-retry.flow.json
  webhook-dispatch.flow.json
```

Relative `src` paths in the manifest are resolved against the manifest file's location. AI harnesses can generate each `*.flow.json` in separate sessions and append entries to the manifest incrementally.

## Build a manifest from disk

Use the agents CLI to scan a directory and generate the manifest automatically:

```bash
vp run flows:manifest -- --dir ./flow-specs --out ./flow-specs/manifest.json
```

This scans all `*.flow.json` files in `--dir` and writes a manifest with `id` and `title` derived from the filename and spec title.

## Inline specs

The `src` field can be an inline spec object instead of a file path. This is useful for embedding small specs directly in the manifest:

```json
{
  "id": "hello",
  "title": "Hello Flow",
  "src": {
    "root": "timeline",
    "elements": {
      "timeline": {
        "type": "FlowTimeline",
        "props": { "title": "Hello" },
        "children": ["start"]
      },
      "start": {
        "type": "TriggerNode",
        "props": { "label": "Start" }
      }
    }
  }
}
```
