# Custom Styling

This example keeps the default Flaier renderer behavior but changes the visual language with scoped CSS variables and `data-*` hooks.

::CustomStyledFlowDemo
::

## Files in this example

- `apps/docs/app/components/CustomStyledFlowDemo.vue`
- `apps/docs/app/plugins/docs-components.ts`
- `apps/docs/public/examples/custom-styling.flow.json`
- `apps/docs/app/components/CalloutNode.vue`

## MDC caveat

`CustomStyledFlowDemo` is rendered directly from markdown, so it must be registered globally by the host app. This is separate from `registerFlaierNodes(...)`, which only registers custom nodes inside the Flaier renderer.

```ts
import CustomStyledFlowDemo from "../components/CustomStyledFlowDemo.vue";

export default defineNuxtPlugin((nuxtApp) => {
  nuxtApp.vueApp.component("CustomStyledFlowDemo", CustomStyledFlowDemo);
});
```

## What is being customized

- `.flaier` theme tokens such as `--color-primary`, `--color-card`, and `--fn-edge-active`
- panel chrome tokens such as `--fn-panel-toggle-bg` and `--fn-panel-toggle-hover-border`
- stable hooks such as `.fn-node[data-node-type="callout"]`
- async edge styling via `.vue-flow__edge.edge-kind-async`

## Why this pattern works

- The theme tokens handle broad visual changes without changing component code.
- The data hooks let you target specific node families when you need a more opinionated style.
- The wrapper keeps overrides local to a single embed instead of restyling every Flaier instance in the app.

Use this approach when you want the shipped renderer behavior and layout, but need a product-specific look and feel.
